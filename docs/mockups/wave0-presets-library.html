Apply a preset and get a normal, editable chain — not a black box.

Scope: the Presets Library inside PresetBrowser (embeddable, UD-2 — no new top-level chrome), plus two U0 insets that ride alongside it: the _mix wet/dry knob on a device card's title bar (existing Knob.tsx convention, not Knob-v3), and the edge-curve enum picker (linear / exponential / logarithmic / s-curve / smoothstep — UD-5, explicitly not a points editor). Every routed preset resolves, on apply, into ordinary devices and ordinary edges — same colors, same matrix, same undo model as anything you'd wire by hand. That transparency guarantee is PK.1's whole point.

Every device card's title bar grows one small knob: the arc geometry is copied from the live Knob.tsx (SIZE 40 · START_ANGLE 135° · SWEEP 270°, gap at the bottom — see frontend/src/renderer/components/common/Knob.tsx:24-29), scaled down for the title bar. It is not the routing-design-suite's proposed Knob-v3 "one circumference" ring — that spec is unbuilt and this packet must not implement or contradict it. At _mix → 0 the device is byte-identical to fully bypassed (this is Packet 4's morph-bypass oracle).

Why the title bar, not the param panel: _mix is the one control every device shares (it's synthesized by routing.py's _mix injection, not a per-effect param), so it gets a fixed slot next to the name instead of competing for space in the effect-specific param list below.
Default 1.0 renders byte-identical to today (no _mix control existed before this packet) — the knob is additive UI over an already-live backend value.

Decisions this mock renders (verbatim, locked)

UD-1 Addressing = instance-UUID end-to-end, TYPE fallback for legacy mappings. Renders as: preset #3's two fx.datamosh instances, one routed, one not. UD-2 Wave 0 builds folders/search inside PresetBrowser, embeddable, no new top-level chrome — B's future browser tree re-hosts it later. UD-3 Full-suite merge gate (not rendered visually — a CI/process decision, not a UI surface). UD-4 Factory packs = subfolders of <Documents>/Creatrix/Presets/; ~/.creatrix/user-library/ stays with B's browser epic. UD-5 Edge-curve v1 = enum picker (existing 4-value CurveType + additive smoothstep); draggable-points editor deferred to K1.

Open design questions flagged, not invented silently

  • Tag filter UI — no PRD locks chip-vs-dropdown-vs-inline-text for tag filtering; this mock guesses multi-select toggle chips AND'd with the search box.
  • "What it does" copy — length, tone, and authorship (hand-written vs. generated from the effect chain) aren't specified; this mock guesses a single ~70-character plain-language sentence per preset.
  • Pack names/roster grouping — the 24 seed presets (PK.3) are named in PRESET-TOP50.md but their grouping into named packs/subfolders isn't decided; the four folder names here (VHS Archive / Cyanotype Series / Long Brake / Chroma Textures) are placeholders for visual variety, not a locked taxonomy.
  • Curve-picker icon glyphs — the 5 little curve sketches are this mock's invention for scannability; only the text labels (linear/exponential/logarithmic/s-curve/smoothstep) are load-bearing per UD-5.
  • Device/route count format — shown as "N devices · N routes" on the card; no PRD specifies the exact string.
